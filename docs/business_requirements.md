# Business Requirements

## 1. Business Context

FinPulse is an end-to-end financial transaction analytics platform designed for a simulated bank or fintech environment.

The platform stores, processes, analyzes, and monitors financial transactions generated by customers through different banking and payment channels such as mobile banking, internet banking, ATM, POS, and bank branches.

The main goal of the project is to support transaction analytics, operational monitoring, fraud detection, risk alert management, and future AI-powered financial analysis.

This project uses synthetic data only. No real customer data, card data, or financial data is used.

## 2. Target Users

* Data Analyst: analyzes transaction volume, transaction amount, failure rate, top customers, top merchants, and channel performance.
* Data Engineer: designs databases, builds data pipelines, creates data warehouses, and processes large-scale financial data.
* Risk Analyst: monitors suspicious transactions, risk scores, fraud rules, and risk alert patterns.
* Fraud Analyst: investigates high-risk transactions and reviews fraud-related alerts.
* Product Manager: evaluates payment channel performance, customer behavior, and transaction trends.
* Operation Team: monitors transaction status, processing issues, and operational exceptions.
* BI Developer: builds dashboards and reports for transaction monitoring and business performance analysis.
* AI Engineer: develops future AI-powered components such as fraud detection models and retrieval-augmented analytics.

## 6. Project Scope

In the current phase, this project focuses on designing the OLTP database for the financial transaction system.

This includes:

* Defining business requirements
* Identifying core data objects
* Designing a Chen-style Enhanced Entity Relationship Diagram
* Defining entities, attributes, primary keys, and relationships
* Modeling superclass/subclass structures
* Modeling weak entities
* Modeling recursive relationships
* Modeling associative relationships
* Creating the SQL Server OLTP schema
* Inserting sample synthetic data
* Writing validation queries

Future phases of the project will extend the platform with:

* Data generation
* Data lake storage
* Batch data processing
* Data warehouse modeling
* Power BI dashboards
* Streaming data pipelines
* Fraud detection machine learning models
* MLOps workflow
* RAG-based financial data assistant

## 7. Business Rules Summary

* Each user must be classified as either a customer or an admin.
* A user cannot be both a customer and an admin at the same time.
* A customer can own multiple accounts.
* Each account must belong to one customer.
* Each account must be managed by one branch.
* Each account must use one currency.
* An account can have multiple cards.
* Each card must belong to one account.
* An account can generate multiple transactions.
* Each transaction must belong to one account.
* A transaction may or may not use a card.
* A transaction may or may not involve a merchant.
* Each transaction must have one transaction type.
* Each transaction must be processed through one channel.
* Each transaction must use one currency.
* Each transaction can have multiple status history records.
* Each status history record must belong to one transaction.
* A transaction can trigger zero or many risk alerts.
* Each risk alert must be based on one fraud rule.
* A risk alert may or may not be reviewed by an admin.
* A transaction can have zero or many fee charges.
* Fee rules are configured by transaction type, channel, and currency.
* An admin can grant permissions to another admin.
